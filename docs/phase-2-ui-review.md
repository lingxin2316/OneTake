## 3. 宽度检查策略

建议在 Chrome DevTools 设备模式下检查：

| 宽度 | 高度建议 | 用途 |
|---:|---:|---|
| 360 | 800 | 小屏 Android 低宽度压力测试 |
| 375 | 812 | iPhone X 类常见宽度参考 |
| 390 | 844 | 当前主流手机宽度参考 |
| 412 | 915 | 大屏 Android 宽度参考 |

检查时页面会在 `<= 520px` 下进入全屏 App 模式，不再显示桌面手机外框，因此宽度检查更接近真实移动端。

---

## 4. 自动/静态检查结论

- 当前未发现 emoji 图标残留；
- 当前无 TypeScript 类型错误；
- 当前无构建错误；
- 主要网格布局使用 `1fr` 自适应列，不依赖固定宽度；
- 底部导航固定 64px，高频页面已预留底部 padding；
- 卡片页辅助操作栏默认关闭，打开后 toast 会上移避让；
- 暂存页三列网格在 360px 下仍有可用宽度，但建议人工确认文字按钮是否显得拥挤。

---

## 5. 人工预览重点

请在 Chrome DevTools 下逐个宽度确认：

- 首页顶部统计卡是否过高或压缩首屏列表；
- 首页展开分组后底部导航是否遮挡最后一行；
- 卡片页主照片卡是否完整可见；
- 右上角钉住按钮是否挡住照片主体；
- 钉住后是否自动进入下一张；
- 撤回 toast 是否避开暂存栏和辅助操作栏；
- 设置页打开辅助操作栏后，返回卡片页底部按钮是否拥挤；
- 暂存筛选页三列网格在 360px 下是否需要改为两列。

---

## 6. 当前判断

当前版本已达到“高保真原型方向稿”标准，可以作为 Android MVP 的视觉蓝本。  
在进入 Android 开发前，还建议做一次人工 Chrome 设备模式检查，重点确认 360px 宽度下的卡片页和暂存筛选页。
